## 行为准则

### 我们的准则

- 使用包容性语言
- 尊重不同的观点和经验
- 优雅地接受建设性批评
- 关注对社区最有利的事情
- 对其他社区成员表示同理心

## 如何贡献

### 报告 Bug

创建 Issue 时请使用 Bug 报告模板（见下方）。

### 提出新功能

在创建 Issue 前请先检查：
- 功能是否符合项目目标
- 是否已有类似的功能请求
- 是否愿意自己实现该功能

### 提交代码

1. Fork 项目仓库
2. 创建特性分支
3. 提交代码
4. 创建 Pull Request
5. 等待代码审查

### 改进文档

- 修正错别字和语法错误
- 添加代码示例
- 翻译文档
- 改进现有文档的可读性

## 开发流程

### 1. 设置开发环境

```bash
# Fork 项目
# 在 GitHub 上点击 Fork 按钮

# 克隆您的 Fork
git clone https://github.com/YOUR_USERNAME/uniadmin.git
cd uniadmin

# 添加上游仓库
git remote add upstream https://github.com/uniadmin/uniadmin.git
```

### 2. 创建分支

```bash
# 同步最新代码
git fetch upstream
git checkout master
git merge upstream/master

# 创建特性分支
git checkout -b feature/your-feature-name
```

### 3. 编写代码

- 遵循 [编码规范](docs/DevelopmentGuide.md#编码规范)
- 编写单元测试
- 更新相关文档

### 4. 测试

```bash
# 运行测试
.\build_and_test.bat

# 确保所有测试通过
```

### 5. 提交代码

```bash
# 添加更改
git add .

# 提交 - 使用清晰的提交信息
git commit -m "feat: 添加产品管理模块

- 实现 ProductDataModule 数据访问层
- 实现 ProductService 业务逻辑层
- 实现 ProductListFrame UI 组件
- 添加单元测试"

# 推送到您的 Fork
git push origin feature/your-feature-name
```

### 6. 创建 Pull Request

在 GitHub 上创建 Pull Request，使用下方模板。

### 7. 代码审查

- 等待维护者审查您的代码
- 根据反馈进行修改
- 所有审查通过后，代码将被合并

## Pull Request 模板

### 标题格式

使用语义化提交标题：

```
<type>(<scope>): <subject>

类型:
- feat: 新功能
- fix: Bug 修复
- docs: 文档更新
- style: 代码格式（不影响功能）
- refactor: 重构
- perf: 性能优化
- test: 测试相关
- chore: 构建/工具相关

范围: 模块名 (如: User, Role, Menu)

主题: 简短描述
```

### 描述模板

```markdown
## 变更类型
- [ ] Bug 修复 (不破坏现有功能的修复)
- [ ] 新功能 (不破坏现有功能)
- [ ] 重大变更 (会导致现有功能不可用的更改)

## 相关 Issue
Closes #xxx
Related to #yyy

## 变更说明
描述您的更改内容和原因。

## 测试
描述您如何测试这些更改：
- [ ] 单元测试通过
- [ ] 手动测试通过
- [ ] 添加了新的测试用例

## 截图/演示
如果是 UI 变更，请添加截图或 GIF 演示。

## 检查清单
- [ ] 代码遵循项目的编码规范
- [ ] 添加了必要的文档
- [ ] 更新了 README
- [ ] 新代码包含单元测试
- [ ] 所有测试通过
- [ ] 提交信息使用语义化格式
```

## 代码审查标准

### 审查流程

1. **自动化检查**
   - CI 编译检查
   - 代码风格检查
   - 单元测试覆盖

2. **人工审查**
   - 功能正确性
   - 代码质量
   - 架构一致性
   - 文档完整性

### 代码质量检查

#### 功能正确性

- ✅ 代码实现了 PR 描述的功能
- ✅ 没有引入破坏性变更（除非已声明）
- ✅ 边界情况得到处理
- ✅ 错误处理完善

#### 代码风格

- ✅ 遵循 [编码规范](docs/DevelopmentGuide.md#编码规范)
- ✅ 命名清晰有意义
- ✅ 代码有适当的注释
- ✅ 没有调试代码或注释代码

#### 架构和设计

- ✅ 遵循 SOLID 原则
- ✅ 模块职责清晰
- ✅ 适当的抽象层次
- ✅ 良好的可扩展性

#### 测试

- ✅ 新功能有单元测试
- ✅ 测试覆盖率 >= 80%
- ✅ 测试用例覆盖边界情况
- ✅ 测试命名清晰

#### 文档

- ✅ 公共 API 有 XML 注释
- ✅ 复杂逻辑有解释注释
- ✅ README 更新（如需要）
- ✅ 变更日志更新（如需要）

### 常见审查意见

#### 必须修复

- 编译错误或测试失败
- 安全漏洞（SQL 注入、XSS 等）
- 破坏性变更（未声明）
- 侵犯版权的代码

#### 建议修复

- 代码风格问题
- 命名不清晰
- 缺少注释
- 可以简化的代码

### 审查通过条件

- 所有自动化检查通过
- 所有"必须修复"项已处理
- 至少一名维护者批准
- 没有待解决的审查反对意见

## Issue 报告模板

### Bug 报告

```markdown
**Bug 描述**
简要描述遇到的问题。

**重现步骤**
1. 访问页面 '...'
2. 点击按钮 '....'
3. 滚动到 '....'
4. 看到错误

**预期行为**
描述您期望发生的事情。

**实际行为**
描述实际发生的事情。

**截图**
如果适用，添加截图帮助说明问题。

**环境信息**
- 操作系统: [Windows 10 / Linux / macOS]
- 浏览器: [Chrome / Firefox / Edge]
- UniAdmin 版本: [1.0.0]
- Delphi 版本: [12 Athens]

**附加信息**
任何其他有助于解决问题的信息。
```

### 功能请求

```markdown
**功能描述**
简要描述您希望添加的功能。

**问题背景**
这个功能解决什么问题？为什么需要它？

**建议的解决方案**
描述您希望这个功能如何工作。

**替代方案**
描述您考虑过的其他解决方案。

**附加信息**
任何其他有助于理解功能请求的信息。
```

### 文档问题

```markdown
**文档位置**
描述有问题的文档位置（文件名或 URL）。

**问题描述**
描述文档中的问题。

**建议的改进**
描述您建议如何改进文档。

**附加信息**
任何其他有助于改进文档的信息。
```

## 开发建议

### 新手友好的 Issues

我们会在 Issue 上标记 `good first issue` 标签，这些 Issue 适合新手贡献。

### 寻求帮助

- 在 Issue 中提问前先搜索是否有类似问题
- 在 Slack/Discord 社区中讨论
- 创建 "Discussion" 而非 Issue

### 设计讨论

大的功能变更应该先讨论再实现：
1. 创建 Issue 描述想法
2. 等待维护者和社区反馈
3. 达成共识后再开始实现
