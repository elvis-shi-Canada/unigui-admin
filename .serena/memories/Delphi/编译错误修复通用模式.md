# Delphi UniGUI 项目编译错误修复通用模式

## 核心原则

**在修复编译错误前，必须理解以下关键点：**
1. 类型的实际定义（接口 vs 类，基类 vs 派生类）
2. 方法的签名和可见性（构造函数参数、返回值类型）
3. 单元依赖关系（类型/函数的来源单元）
4. 框架的设计模式（数据模块生命周期、上下文注入方式）

---

## 模式 1：数据模块创建与上下文注入

### 问题场景
创建继承自 `TUniDataModule` 的子类实例时，需要同时处理所有者管理和上下文注入。

### 错误模式
```pascal
// ❌ 错误：混淆了所有者参数和上下文参数
FDataModule := TMyDataModule.Create(Context);  // Context 是 IExecutionContext，不是 TComponent
```

### 正确模式
```pascal
// ✅ 正确：分离所有者管理和上下文注入
FDataModule := TMyDataModule.Create(Self);     // Self 作为所有者
FDataModule.SetContext(Context);               // 单独注入上下文
```

### 设计原则
- **所有者管理**：Delphi 的组件所有者机制要求 `Create(AOwner: TComponent)`
- **上下文注入**：使用 `SetContext` 方法注入业务上下文，而非通过构造函数
- **生命周期**：所有者负责释放，上下文负责业务逻辑

### 适用范围
所有继承自 `TUniDataModule` 或类似基类的数据模块创建场景

## 模式 2：连接对象类型转换

### 问题场景
在多层架构中，基类返回抽象连接类型，但具体方法需要具体连接类型。

### 错误模式
```pascal
// ❌ 错误：类型不匹配
procedure SetConnection(const Value: TFDCustomConnection);
...
MyDialog.SetConnection(ModelAdmin.Connection);  // TFDCustomConnection → TFDConnection 不兼容
```

### 正确模式
```pascal
// ✅ 正确：显式类型转换
MyDialog.SetConnection(TFDConnection(ModelAdmin.Connection));
```

### 设计原则
- **类型层次**：理解 `TFDCustomConnection`（基类）与 `TFDConnection`（实现类）的关系
- **显式转换**：当确定运行时类型时，使用显式转换
- **安全检查**：必要时使用 `as` 操作符进行安全类型检查

### 适用范围
所有涉及 FireDAC 连接对象的跨组件传递场景

## 模式 3：接口类型 vs 类类型

### 问题场景
误以为存在接口类型，但实际定义的是类类型。

### 错误模式
```pascal
// ❌ 错误：假设存在接口
FManager: IMyDataModule;  // 实际定义的是 TMyDataModule 类
```

### 正确模式
```pascal
// ✅ 正确：使用实际定义的类型
FManager: TMyDataModule;
```

### 设计原则
- **验证定义**：使用前通过搜索确认类型的实际定义（接口 vs 类）
- **命名约定**：Delphi 中接口以 `I` 开头，类以 `T` 开头
- **不要假设**：不要根据命名习惯假设，要查看实际代码

### 适用范围
所有跨模块引用类型声明场景

## 模式 4：方法返回值类型理解

### 问题场景
混淆不同对话框的 `Execute` 方法返回值类型。

### 错误模式
```pascal
// ❌ 错误：假设 Execute 返回 ModalResult
if LDialog.Execute = mrOK then  // Execute 可能返回 Boolean
```

### 正确模式
```pascal
// ✅ 正确：根据实际声明检查返回值
// 模式 A：返回 Boolean
if LDialog.Execute then
begin
  // 成功处理
end;

// 模式 B：返回 ModalResult（标准 VCL 窗体）
if LDialog.ShowModal = mrOK then
begin
  // 成功处理
end;
```

### 设计原则
- **区分 Execute 和 ShowModal**：自定义对话框通常返回 Boolean，标准窗体返回 ModalResult
- **查看声明**：检查方法的实际返回类型
- **一致性**：同一项目中保持统一的对话框模式

### 适用范围
所有对话框调用场景

## 模式 5：基类方法可见性设计

### 问题场景
子类需要调用基类的 protected/private 方法。

### 错误模式
```pascal
// ❌ 错误：方法在 private 部分，子类无法访问
TBaseFrame = class
private
  procedure BtnEditClick(Sender: TObject);  // 子类无法调用
```

### 正确模式
```pascal
// ✅ 正确：将需要被子类调用的方法放在 protected
TBaseFrame = class
protected
  procedure BtnEditClick(Sender: TObject);  // 子类可以调用
```

### 设计原则
- **可见性规则**：
  - `private`：仅本类可访问
  - `protected`：本类和子类可访问
  - `public`：所有代码可访问
- **设计意图**：基类应明确哪些方法可供子类扩展或调用
- **重构时机**：当子类需要调用基类方法时，调整可见性

### 适用范围
所有继承体系中的方法可见性设计

## 模式 6：标准函数 vs 假设的方法

### 问题场景
假设字符串或对象有某个方法，但实际是独立函数。

### 错误模式
```pascal
// ❌ 错误：假设的方法
if not LString.IsValidIdentifier then
  raise Exception.Create('Invalid');
```

### 正确模式
```pascal
// ✅ 正确：使用标准函数
if not System.SysUtils.IsValidIdent(LString) then
  raise Exception.Create('Invalid');
```

### 设计原则
- **验证 API**：使用前通过文档或搜索确认方法是否存在
- **Delphi 风格**：Delphi 倾向于使用独立函数而非对象方法
- **命名空间**：使用完整的命名空间限定（如 `System.SysUtils`）

### 适用范围
所有字符串处理、类型转换等工具函数调用场景

## 模式 7：UniGUI 应用程序对象访问

### 问题场景
访问 UniGUI 全局应用程序对象时使用错误的命名空间。

### 错误模式
```pascal
// ❌ 错误：错误的命名空间
LDialog := TMyDialog.Create(UniGUIApplication.UniApplication);
```

### 正确模式
```pascal
// ✅ 正确：直接使用全局对象
LDialog := TMyDialog.Create(UniApplication);
```

### 设计原则
- **全局对象**：`UniApplication` 是全局变量，无需命名空间前缀
- **单元引用**：确保引用了 `uniGUIBaseClasses` 或 `uniGUIApplication`
- **一致性**：遵循项目中其他代码的访问方式

### 适用范围
所有访问 UniGUI 全局对象的场景

## 模式 8：字段声明完整性

### 问题场景
使用了未声明的私有字段。

### 错误模式
```pascal
// ❌ 错误：缺少字段声明
type
  TMyForm = class
  private
    FQuery: TFDQuery;
    // 缺少 FSelectedID 声明
  ...
procedure TMyForm.DoSomething;
begin
  FSelectedID := 123;  // E2003 错误
```

### 正确模式
```pascal
// ✅ 正确：完整声明所有字段
type
  TMyForm = class
  private
    FQuery: TFDQuery;
    FSelectedID: Integer;  // 添加声明
```

### 设计原则
- **先声明后使用**：所有字段必须在 type 部分声明
- **完整性检查**：编译错误提示未声明标识符时，首先检查是否遗漏字段
- **命名规范**：使用 `F` 前缀标识私有字段

### 适用范围
所有类/窗体的字段声明场景

## 模式 9：标准单元引用模板

### 问题场景
缺少必要的单元引用导致类型/函数未定义。

### 错误模式
```pascal
// ❌ 错误：缺少关键引用
uses
  System.SysUtils, System.Classes;
  // 缺少 System.UITypes, Vcl.Dialogs, uniGUIForm 等
```

### 正确模式
```pascal
// ✅ 正确：标准引用顺序
uses
  // System 单元
  System.SysUtils, System.Classes, System.Variants, System.UITypes, System.Types,
  // Data 单元
  Data.DB, FireDAC.Comp.Client,
  // VCL 单元
  Vcl.Forms, Vcl.Controls, Vcl.Dialogs,
  // UniGUI 基础单元
  uniGUIBaseClasses, uniGUIClasses, uniGUIForm, uniGUIFrame,
  // UniGUI 控件单元（按需添加）
  uniEdit, uniButton, uniLabel, uniPanel, ...
  // 项目单元
  UniContext, UniPlugin.Types, ...;
```

### 设计原则
- **分组排序**：System → Data → VCL → UniGUI → 项目单元
- **最小依赖**：只引用实际使用的单元
- **完整性**：确保所有使用的类型/函数都有对应的单元引用

### 适用范围
所有 Delphi 单元的 uses 部分设计

## 模式 10：Record 字段名匹配

### 问题场景
使用 record 类型时假设字段名，但实际字段名与假设不同。

### 错误模式
```pascal
// ❌ 错误：假设的字段名
type
  TPluginInfo = record
    PluginID: string;
    DisplayName: string;
  end;
  
var
  Info: TPluginInfo;
begin
  Info.Name := 'TestPlugin';  // E2003 错误：实际字段是 PluginID，不是 Name
```

### 正确模式
```pascal
// ✅ 正确：对照实际声明使用字段
var
  Info: TPluginInfo;
begin
  Info.PluginID := 'TestPlugin';  // 使用实际定义的字段名
```

### 设计原则
- **查看定义**：使用 record 前，先查看实际的字段声明
- **不要推断**：不要根据类似类型推断字段名
- **特别注意**：区分 `TPluginClassInfo` vs `TPluginInfo` 等相似类型

### 适用范围
所有 record 类型的字段访问场景

## 模式 11：Dispose vs Free 的正确使用

### 问题场景
混淆 `Dispose` 和 `Free` 的用法，对类实例使用 `Dispose`。

### 错误模式
```pascal
// ❌ 错误：对类实例使用 Dispose
FDataScopes.Dispose;  // E2003 错误：Dispose 未定义
FLock.Dispose;
```

### 正确模式
```pascal
// ✅ 正确：类实例使用 Free
FDataScopes.Free;
FLock.Free;

// ✅ 正确：Dispose 仅用于 New 分配的指针
var
  P: PInteger;
begin
  New(P);
  Dispose(P);  // 正确用法
end;
```

### 设计原则
- **TObject 派生类**：使用 `Free` 或 `FreeAndNil`
- **指针类型**：`New` 分配的内存使用 `Dispose` 释放
- **从其他语言迁移**：将 `.Dispose()` 替换为 `.Free`

### 适用范围
所有对象释放和内存管理场景

## 模式 12：方法参数数量匹配

### 问题场景
调用方法时参数数量与声明不匹配。

// ❌ 错误：参数数量不匹配
procedure AddDependency(const FromPluginID, ToPluginID, MinVersion: string);

### 正确模式

### 设计原则
- **查看声明**：调用前检查方法的完整签名
- **可选参数**：使用默认值显式传递可选参数
- **重构同步**：修改方法签名后，立即更新所有调用点

### 适用范围
所有方法调用场景

## 模式 13：TUniForm/TUniFrame 类型引用

### 问题场景
使用 TUniForm 或 TUniFrame 时缺少必要的单元引用。

### 错误模式
```
E2086: Type 'TUniForm' is not yet completely defined
E2003: Undeclared identifier: 'TUniForm'
E2003: Undeclared identifier: 'TUniFrame'
```

### 正确模式
```pascal
// ✅ 正确：TUniForm 需要 uniGUIForm 单元
uses
  uniGUIForm;  // TUniForm 定义在此单元

// ✅ 正确：TUniFrame 需要 uniGUIFrame 单元
uses
  uniGUIFrame;  // TUniFrame 定义在此单元
```

### 设计原则
- **禁止前向声明后立即使用**：
  ```pascal
  // ❌ 错误
  TUniForm = class;
  TFormClass = class of TUniForm;  // E2086
  
  // ✅ 正确
  uses uniGUIForm;
  TFormClass = class of TUniForm;
  ```
- **修复顺序**：先修复类型单元，再修复实现单元，最后修复窗体单元

### 适用范围
所有继承或使用 TUniForm/TUniFrame 的场景

## 快速诊断流程

当遇到编译错误时，按以下顺序诊断：

### 步骤 1：识别错误类型
```
E2003: Undeclared identifier → 类型/函数未定义
E2010: Incompatible types → 类型不匹配
E2086: Type not yet completely defined → 类型定义不完整
E2065: Unsatisfied forward declaration → 方法未实现
```

### 步骤 2：查找定义位置
```bash
# 搜索类型定义
grep -r "TTypeName = class" src/
grep -r "IInterfaceName = interface" src/

# 搜索函数/过程声明
grep -r "function FunctionName" src/
```

### 步骤 3：检查单元引用
- 确认 uses 部分包含定义该类型/函数的单元
- 检查是否在 interface 部分引用（而非 implementation）

### 步骤 4：验证方法签名
- 检查构造函数参数类型
- 检查返回值类型
- 检查参数数量和类型

### 步骤 5：应用修复模式
- 根据错误类型选择对应的修复模式
- 最小化修改，一次只修复一个问题
- 重新编译验证

## 架构原则

### 1. 依赖注入模式
- 使用 `SetContext` 注入业务上下文
- 构造函数只接受 `TComponent` 所有者
- 数据模块通过所有者自动管理生命周期

### 2. 类型安全模式
- 理解类型层次关系（基类 vs 派生类）
- 必要时使用显式类型转换
- 避免隐式类型转换导致的错误
- 区分接口类型（I 前缀）和类类型（T 前缀）

### 3. 可见性设计
- 基类的扩展点使用 protected
- 内部实现细节使用 private
- 公共 API 使用 public

### 4. 单元组织
- 标准引用顺序确保依赖清晰
- 避免循环依赖
- 最小化依赖减少编译时间
- TUniForm/TUniFrame 必须引用对应单元

### 5. 内存管理
- TObject 派生类使用 Free/FreeAndNil
- 指针类型使用 New/Dispose
- 严格区分两种内存管理方式

### 6. Record 使用规范
- 使用前查看实际字段定义
- 不根据类似类型推断字段名
- 区分相似的 record 类型

### 7. 方法调用规范
- 调用前检查完整签名
- 修改签名后立即更新所有调用点
- 可选参数显式传递默认值
